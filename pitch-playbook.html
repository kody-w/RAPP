01The Problem: Agent Fragmentation

Across Microsoft, teams are independently building agent tooling that is complex, redundant, and unteachable. The more we adopt external ecosystems that go in the same direction, the worse it gets. Only a handful of people can use any single tool. None of them share agents with each other.

Core tensions
  • "Yeah, we built that too" — heard dozens of times over two years. Teams build redundant tools that won't scale, but use them because they came up with them.
  • Complexity as a moat — if people can't learn the tools, it's just floating egos. The tools exist to serve their creators, not the org.
  • Tool balkanization delays the real unlock — people forget what comes after we get good tools people can build on. That's where the AI value comes from.
  • The two-year prediction — everyone will eventually need a quick way to share agents. They could adopt RAPP now or clean up the mess later.

This is what RAPP is trying to stop. It works for everyone and is easy to learn and teach. It's trying to prevent another AI winter and just jump over it.

02Why RAPP Solves It

The sacred tenet

One file = one agent. No exceptions.

A single Python file contains the documentation, the contract, and the code. A 14-year-old can ship their first agent in minutes. An enterprise can ship the same file to 500,000 seats.

  • Three tiers, zero modification. The same file runs on a laptop (local Flask server), in Azure Functions, and in Copilot Studio on Teams — unchanged.
  • Shareable by design. Agents install with a file drop. There's a registry, a store, and agents can be compressed into a speakable 7-word phrase.
  • Already working. RAPP has a frozen v1 spec, a one-line installer, a live agent store, and a pipeline that generates agents from natural language.
  • Engine, not experience. Infrastructure that stays light so capabilities compound. The opposite of every framework that ships complexity and calls it power.

07Agent Management: Enable/Disable Toggle

The brainstem now supports per-agent enable/disable toggles. Files stay on disk — they're just skipped during load. This lets you curate which swarms are active for any given demo or workflow.

How it works

  • Backend: .agents_disabled.json in the agents directory tracks disabled filenames
  • API: POST /agents/<filename>/toggle flips enabled/disabled state
  • Load: load_agents() skips files listed in the disabled set
  • UI: Green toggle switch next to each agent in the panel

Visual

execbrief_agent.py — ExecBrief
hacker_news_agent.py — HackerNews
bookfactory_agent.py — BookFactory

One click to toggle. No file deletion. Non-destructive and reversible. State persists across restarts.

08The Ultimate Dropper: Teams + Virtual Brainstem

This is the closer. After the demo, you hand the executive the agent itself — not a slide deck, not a doc, not a link to a repo. The actual agent as a single .py file. They drop it into a browser and it works.

The play

Post the file. Post the link. They're running it in 30 seconds.

  1. Export — from the agents panel, click Export on execbrief_agent.py. You now have the singleton file.
  2. Post to Teams — drop the .py file directly into the Teams chat with the executive. One file, 489 lines, everything inlined.
  3. Post the link — paste the virtual brainstem URL alongside it: 
    https://kody-w.github.io/RAPP/rapp_brainstem/web/
  4. They open & drop — the executive opens the virtual brainstem in their browser, opens the Agents drawer, and drops execbrief_agent.py onto the drop zone. It loads instantly.
  5. They run it — they can see the agent, its metadata, its capabilities. To execute the full pipeline, they run the one-liner to start a local brainstem (the tether lights up green) and the browser routes execution to local Python.
